Architecture

DVL is organized like a database engine, because it is one: an authoring frontend produces a spec, the spec is rewritten and planned into a dataflow of operators, an optimizer decides where each operator runs, and a scheduler executes the graph to a fixpoint. This page describes the architecture the codebase is converging on — the refactor's target — so that each concept has exactly one home. (Rationale and the migration plan live in docs/2026-07-13-codebase-reorg-review.html.)

The pipeline

author ──▶ rewrite ──▶ resolve ──▶ plan ──▶ optimize ──▶ bind ──▶ execute ──▶ emit
builder     axes→plots   catalog     OpSpec    substrate,   impls    SCC          scene
+ spec      (desugar)    lookup      graph     fusion, DCE           fixpoint     → SVG
Stage What it does DB analog
author the fluent builder populates a Spec — nodes, plots, channel expressions, scale/layout instances, and cells (solver variables) parser / binder
rewrite spec→spec desugaring: axes (and legends) become ordinary plots over tick relations rewriter
resolve every by-name component reference is resolved against the registries once; downstream never touches a registry catalog lookup
plan each node's plots compile to operator descriptions (OpSpec: id, reads, writes, payload) — one logical dataflow graph, nothing executed logical planner
optimize rule passes choose each edge's substrate (SQL or JS), insert conversions, fuse, and eliminate dead ops optimizer
bind descriptions become runnables: payload kind → registered handler physical plan
execute the scheduler condenses the graph into SCCs and runs them in topological order; cyclic components iterate to a residual fixpoint (this is how measurement-driven sizing converges) executor
emit placed marks materialize into a scene graph; a pen walks it into SVG client

Two supporting systems sit beside the pipeline: the data layer (backend adapter, dialect, SQL fragments with fold/emit composition — the only place SQL text is produced) and the debug harness (a sink that records every pass, statement, and solver iteration into dvl_* tables, off by default at zero cost).

The attribute type system

The engine's central discipline is that every column and cell carries a type with three orthogonal axes:

  • carrier — the physical representation (double, varchar, …). The database sees only this.
  • space — the semantic world the value lives in: data (source values and identity keys) · domain (fitted metadata about data: envelopes, category indexes) · relative (normalized [0, 1] frame positions) · pixel (placed geometry, extents, demands) · style (color/shape/text/opacity — final values the pen consumes) · scene (the artifact).
  • geom — spatial shape (coord⟨x|y⟩, len⟨x|y⟩, area), present only in the relative and pixel spaces.

A double can be data, domain, relative, or pixel — carrier and space never constrain each other. The pipeline is the space transitions:

data ──fit──▶ domain ──apply──▶ relative ──affine──▶ pixel ──emit──▶ scene
                                            ▲                    │
                                            └──── measure ◀──────┘   (demands re-enter the solver)

Mixed-space arithmetic is ill-typed except through registered coercions: the affine is the unique relative→pixel coercion (it flips space and preserves geom), exactly as materialize/inject are the unique SQL⇄JS coercions on the substrate axis. Planning is demand-driven insertion at mismatches: a positional channel demands pixel, the bound expression yields relative, so the planner inserts the affine — the same mechanism that inserts substrate crossings.

Operators

Scales, layouts, tick generators, and measurement providers share one component contract: an Operator is a catalog entry carrying dual execution implementations (sql and/or js) plus a declared signature over attribute spaces. Signatures are row-polymorphic — they name the columns an operator requires and the columns it adds; everything else flows through:

train   : {v: data | ρ}                        → {lo, hi: domain(over v)}
apply   : {v: data | ρ} × domain(over v)       → {out: relative⟨geom⟩ | v | ρ}
refine  : {x: pixel⟨g⟩ | ρ} (× extents)        → {x′: pixel | ρ}
measure : {…pixel geoms… | ρ}                  → pixel scalars (demands)

The three layout classes fall out of the signatures rather than being a taxonomy: produce (data → relative: scales, bandX, stackY, force, treemap), refine (pixel → pixel, post-placement: fitText, grid packing, size-aware layouts), and measure (pixel → data: text extents, bounding boxes feeding the demand loop). A domain, when an operator trains one, owns its derivation — "a train without a domain" is unrepresentable.

The plan and its payloads

The dataflow graph's nodes are OpSpecs — {id, reads, writes, payload} over cells (the graph's edges: scalar solver variables, or relation handles). The payload vocabulary is small and closed:

Kind Signature Role
transform relations → relation; output attrs derived from a serializable relational-algebra tree every framework data transform: derived columns, joins (via, nest keys), aggregates, the affine
call whatever the component declares the extension boundary: scale/layout train & apply, ticks, measurement, decisions
solve pixel scalars → pixel scalars one node's box constraints (propagation or Cassowary), re-run inside fixpoints
emit pixel relations → scene the render sink
crossing attr-identity, substrate conversion SQL⇄JS materialization

Only transform is structurally rewritable by the optimizer (fusion, pushdown); the rest participate in substrate and liveness decisions but are semantically opaque.

Execution

  • Cells are the dataflow's edges and the constraint system's variables at once. Operators read and write cells; the scheduler orders execution purely from those declarations.
  • Cycles are features. Measurement→demand→arrange→placement→measurement is a genuine loop; the scheduler runs each strongly-connected component as a Gauss–Seidel fixpoint with a residual threshold, then seals its cells.
  • Two substrates, one meaning. Every algebra-built operator gets SQL and JS implementations from the same description; opaque algorithms (d3-force, treemap) declare what they can run on. The optimizer picks per edge; a keyed (per-panel) edge over a non-keyed implementation runs the correlated per-group definition automatically, in the implementation's own substrate.

Late-bound plot references ($… placeholder reads)

The plan walk visits nodes parent-first (a child's compile needs the parent's host cells), but a plot may reference a plot the walk hasn't reached yet: a top-level edge layer whose frame: or position via() points at a mark nested inside a template child. At emission time the referent's cells don't exist, so the compiler writes a placeholder read — a string in the reads slot that names a plot's build product instead of a cell:

Placeholder Resolves to Emitted by
$plot-keyed:<plotId> the referent plot's post-cell-key marks (the relation carrying dvl_cell) — what a frame: borrow joins the frame-assignment borrow
$ref:<plotId> the referent plot's final placed marks (post-affine pixels) — what a position via() joins cross-node position vias

General semantics. A placeholder is a promise resolved exactly once: after the full node walk, compileSpec patches every placeholder read to the built plot's cell and throws if one cannot be satisfied (unknown plot, or a frame: referent that carries no dvl_cell because it was never nested). The invariants that keep the mechanism honest:

  • placeholders appear only in reads, never in writes — a plot may depend on a later plot, but only the owning pipeline writes its cells;
  • they name plots, not cells — the indirection is "this plot's keyed/placed relation", chosen by the patch, so pipelines stay free to allocate cells incrementally;
  • they are gone before the optimizer or scheduler ever see the graph — downstream, every read is a real cell, and an unresolved placeholder is a compile error, never a runtime mystery;
  • cyclic references (A vias B while B vias A) still resolve — patching is name lookup, not execution order; the scheduler's SCC machinery owns any resulting cycle.

To add a new build product to the vocabulary: emit $plot-<name>:<plotId>, expose the cell on PlotBuild, and add its arm to the patch loop in executor/compile-spec.ts with a loud error for the unsatisfiable case.

Module map (target)

src/lib/dvl/
├── types/        the attribute system: Attr = carrier ⊥ space ⊥ geom; typed schemas; ids
├── data/         storage seam: backend, dialect, database, SQL fragments (fold/emit)
├── ir/           the authored spec: nodes, plots, channel exprs, instances, cells
├── api/          the fluent builder + validation (the authoring frontend)
├── algebra/      the relational op algebra + its SQL/JS lowering + Relation/crossings
├── operator/     the component contract: Operator, op(), bindings, registries + resolve
├── plan/         OpSpec, typed payloads, spec→plan compiler, rewrite + plan passes
├── layout/       the spatial engine: frame/affine, arrange (constraints + solvers), measure
├── optimizer/    rule passes: substrate choice, crossings, fusion, DCE
├── executor/     handler dispatch + the SCC/fixpoint scheduler
├── components/   the pluggable instances: scales/, layouts/, marks/, ticks/
├── render/       scene graph, pens (SVG), rasterization
└── debug/        the recording sink, dvl_* schema, self-visualizations

The rule the map enforces: one concept, one module — to understand scales you read operator/ + components/scales/; to understand placement you read layout/; to understand "where does my SQL come from" you read algebra/ + data/. Marks stay deliberately dumb (channels + a draw function); everything smart about position is an operator.
