force

force places nodes with a force-directed simulation and emits an x, y per node, normalized to [pad, 1−pad] (the frame affine turns them into pixels). It takes an edges table — force('edges') — whose endpoint columns come from the declared join (or an on override), never a name convention; links pull connected nodes together, charge pushes all nodes apart. There is no trained domain: the positions carry no data-space meaning, so nothing is tickable.

It is dual-substrate: JS wraps seeded d3-force (stop, tick to convergence); SQL runs the attraction/repulsion/gravity iteration as one recursive DuckDB query (WITH RECURSIVE … USING KEY). Both honor the same contract — normalized positions, deterministic per substrate — but they are different simulations, so the exact coordinates differ between JS and SQL.

const v = canvas(640, 460);
v.dot('members', {
  ...force('follows').pick('x', 'y'),
  r: 8, fill: ordinal('community'),
});
return v;

A node-link scatter of members: force('follows') reads the follows edge table, whose src/dst columns are FK-joined to members.id, and solves each member's position. (Draw the edges themselves with a link layer — see below.)

Options

force(edges) takes the marks (the nodes, whose id column is the primary input) plus an aux edges relation named as the argument. Its endpoint columns default to the declared refs (edges.src → marks.id, edges.dst → marks.id); an on override on the layer changes them. Edges to nodes absent from a panel are dropped (so a per-panel run ignores cross-panel edges). In a template it runs per cell by default — one simulation per panel, each normalized to its own [pad, 1−pad] (perCell: true).

Input Type Meaning
id (marks) int the node identity column, matched against edge endpoints
edges table the edge relation; src/dst (or the declared refs) are its endpoint columns

.pick() outputs:

Output Space Meaning
x relative coord x node x, min-max normalized into [pad, 1−pad] (pad = 0.05)
y relative coord y node y, likewise

Bind by spreading: ...force('follows').pick('x', 'y'). An empty edge set gives pure repulsion; a single node centers at 0.5.

Examples

Plain node placement (as above) — nodes coloured by community:

const v = canvas(640, 460);
v.dot('members', { ...force('follows').pick('x', 'y'), r: 8, fill: ordinal('community') });
return v;

Nodes and edges — draw the links first (underneath), each endpoint borrowing the solved node position through a via on the FK. The nested linear scales in the node plot are unnecessary here; the link's endpoints come straight from the force solve:

const v = canvas(640, 460);
const nodes = v.dot('members', { ...force('follows').pick('x', 'y'), r: 8, fill: ordinal('community') });
v.link('follows', {
  ...nodes.via(eq('follows.src', 'members.id')).pick({ x1: 'x', y1: 'y' }),
  ...nodes.via(eq('follows.dst', 'members.id')).pick({ x2: 'x', y2: 'y' }),
  stroke: '#b6bdc9',
});
return v;

Force rects — the nodes are rectangles hosting nested plots. Force solves the rect's top-left; w/h are pixel literals so every node is a little panel. The link endpoints want the middle, so each via computes a center (f.x + 46, f.y + 32 — half of w/h). Each rect nests that member's outgoing follows on the FK eq('src','id'):

const v = canvas(760, 420);
const nodes = v.rect('members', {
  ...force('follows').pick('x', 'y'),
  w: 92, h: 64, fill: '#ffffff', stroke: ordinal('community'), strokeWidth: 2,
});
const center = { resolve: { cx: { fn: 'avg', arg: expr`f.x + 46` },
                            cy: { fn: 'avg', arg: expr`f.y + 32` } } };
v.link('follows', {
  ...nodes.via(eq('follows.src', 'members.id'), center).pick({ x1: 'cx', y1: 'cy' }),
  ...nodes.via(eq('follows.dst', 'members.id'), center).pick({ x2: 'cx', y2: 'cy' }),
  stroke: '#b6bdc9',
});
const cell = nodes.nest('follows', { on: eq('src', 'id') });
cell.dot({ x: linear('dst', { axis: false }), y: linear('weight', { axis: false }), r: 2.5, fill: '#4e79a7' });
return v;

Force facets — the partition model. eqX lays out one panel per community; a nested force runs per panel (its own [pad, 1−pad]). Because force is perCell, each community's nodes are solved independently inside their panel:

const communities = db.table('members').groupby(['community'], {});
db.declareFK({ fromTable: 'members', fromCols: ['community'],
               toTable: communities.name, toCols: ['community'] });
const v = canvas(760, 360);
const panel = v.rect(communities, { ...eqX('community').pick('x', 'w'), y: 12, h: 330, fill: '#f5f6fa' });
const np = panel.nest('members', { on: eq('community', 'community') }).dot({
  ...force('follows').pick('x', 'y'),
  r: 6, fill: ordinal('community'),
});
// within-panel edges: σ derived from the FK catalog (both endpoints must be in the panel)
panel.link('follows', {
  ...np.via(eq('follows.src', 'members.id')).pick({ x1: 'x', y1: 'y' }),
  ...np.via(eq('follows.dst', 'members.id')).pick({ x2: 'x', y2: 'y' }),
  stroke: '#b6bdc9',
});
return v;

Owner-semantics underlay — a top-level link layer with frame: src opts out of the per-panel partition, drawing every edge in its source node's panel (so a cross-community edge reaches between panels). Authored first, it sits underneath the within-panel layer above:

const communities = db.table('members').groupby(['community'], {});
db.declareFK({ fromTable: 'members', fromCols: ['community'],
               toTable: communities.name, toCols: ['community'] });
const v = canvas(760, 360);
const panel = v.rect(communities, { ...eqX('community').pick('x', 'w'), y: 12, h: 330, fill: '#f5f6fa' });
const np = panel.nest('members', { on: eq('community', 'community') }).dot({
  ...force('follows').pick('x', 'y'), r: 6, fill: ordinal('community'),
});
const src = np.via(eq('follows.src', 'members.id'));
v.link('follows', {   // owner semantics: every edge, assigned to its src's panel
  ...src.pick({ x1: 'x', y1: 'y' }),
  ...np.via(eq('follows.dst', 'members.id')).pick({ x2: 'x', y2: 'y' }),
  stroke: '#dfe3ec',
}, { frame: src });
return v;
