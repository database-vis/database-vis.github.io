Layouts

Layouts are operators that compute arrangements. Most are multi-output: bind them by spreading .pick() into the channel object — ...bandX("cat").pick("x", "w") — or rename with the object form: .pick({ x1: "x", y1: "y" }). Outputs are normalized [0, 1]; the frame affine turns them into pixels.

The four classes

Every layout is one of four kinds, told apart by what it reads and what it writes — the data-flow direction:

  • Produce — data → normalized positions. The common case: a train/apply operator reduces the marks to a fit (its trained domain), then joins that fit back per row to emit normalized [0, 1] channel columns (bandX, barY, stackY, eqX, proportional*). Iterative producers (force, treemap) emit the same normalized outputs from a custom plan (a recursive query in SQL). Bound to channels with .pick(); the frame affine turns [0, 1] into pixels.
  • Refine — placed pixels → adjusted pixels. Runs after placement, reading the marks' already-solved geometry: fitText reads each label's box (w/h) and the string it draws to solve a font size; grid-stack packs items into the content box. Refine outputs are pixels — they never pass back through the affine.
  • Resolver — a matched SET → one value or shape per row. Not a placement stage at all: a resolver rides a via() join, reducing each anchor row's set of matched referent rows to a single derived quantity. Scalar resolvers mint a number (mode, range, extentUnion); geometry resolvers mint a shape — convex/bbox reduce a borrowed point set to a polygon the hull mark draws. Authored as a via() resolution + a mark, not a .pick() operator (see hull).
  • Measure (internal) — placed geometry → size demands. The reverse direction: a measure reports how much room a plot or label needs, feeding the constraint solver's demand loop (an axis gutter widens to fit its measured labels; a panel grows to its measured content). Not exposed in the authoring API — there is no builder verb; the engine wires the built-in providers itself (the label-extent provider estimate, or a canvas measureText in the browser, and the plot bounding-box op). Listed here for the complete picture, not to author.

Three of the four close the frame's fixpoint: produce emits normalized coordinates → the affine places them → refine nudges the placed pixels → measure reads the result back into size demands that re-solve the arrangement, until it settles (see Architecture and The frame). Resolver is orthogonal — a data reduction over a join, not a placement stage in that loop.

Built-in layouts

Layout Outputs What it does
bandX x, w categorical → evenly spaced bands (a vertical bar's x); options paddingInner, paddingOuter, padding, align
bandY y, h categorical → evenly spaced bands (a horizontal bar's y); the y-mirror of bandX, same options
barY y, h value → a vertical bar from the baseline up (a bar's y); option zero: true anchors the domain at 0 (otherwise the domain is the data extent and the min-value bar has height 0)
barX x, w value → a horizontal bar from the baseline out (a bar's x); the x-mirror of barY, same zero: true option
stackY y, h { weight, group, order } → vertically stacked segments per group (a percent-stack, normalized to the box)
stackX x, w { weight, group, order } → horizontally stacked segments per group; the x-mirror of stackY
pitchY y, h fixed-pitch vertical bands — one band per row, ordered; height is a constant pitch independent of the box (legend items, list rows). Pure windowed derive, so it stays in SQL (unlike grid-stack)
pitchX x, w fixed-pitch horizontal bands — one band per row, ordered; width is a constant pitch independent of the box; the x-mirror of pitchY (horizontal legend/list row)
eqX x, w distinct keys → equal horizontal split (the simplest column-facet driver)
eqY y, h distinct keys → equal vertical split (row facets); the y-mirror of eqX
proportionalX / proportionalY x, w / y, h segments sized proportionally to a value column
treemap x, y, w, h squarified treemap over a size column — dual-substrate: d3-hierarchy in JS, or the squarify state machine as one recursive SQL query; the two agree (squarify is deterministic)
force x, y force-directed positions; takes an edges table (force("edges") — endpoint columns come from the declared join, or an on override, never a name convention) — dual-substrate: seeded d3-force in JS, or a recursive in-database simulation (WITH RECURSIVE … USING KEY); same contract (normalized, deterministic per substrate), different exact coordinates
fitText fontSize, rotate, lines fits text into boxes — runs after placement (a refine-class layout). Inputs are inferred from the mark's channels: a bare fitText() sizes exactly the string the mark draws (its text channel, literal or column) against its w/h box — name an input only to override ({ text: 'other_col' }). Defaults do the whole job: each label solves against its own box, rotation is tried (−90° when vertical fits larger), and the size is uncapped — the box itself bounds the solve ("as big as fits"). Opt-outs mix into the same bag (input and option keys are disjoint): shared: true (ONE size across all labels — uniform typography), rotate: false, a finite maxFontSize ceiling, minFontSize, padding, wrap
grid-stack x, y, w, h packs items into the available box, row- or column-major (pixel outputs)
hull path (polygon) a shape around a matched point set — not a .pick() layout but a mark + via geometry resolver: pts.via(on("g"), { resolve: "convex" }).pick("path") reduces the borrowed (placed) points to a polygon (inputs default to x/y, output to path). convex — SQL push-down via DuckDB spatial (ST_ConvexHull), JS reduce as the escape hatch — or bbox (JS)

Substrates. An operator may carry a SQL impl, a JS impl, or both; when both exist the engine keeps the data where it already is (SQL marks stay in the database — no materialize round-trip). Even iterative layouts fit in SQL: force and treemap run as single recursive queries (DuckDB's WITH RECURSIVE, with USING KEY for force's keyed per-node state).

Adding your own layout

A layout is an operator, authored with the same op({ … }) factory as scales and dropped into components/layouts/. The class decides what you supply:

  • Produce: declare a trained domain (continuous/categorical) + an apply returning the normalized outputs — op() wires the train→apply subgraph. For an iterative producer, give a custom plan(io) emitting a recursive RelOp instead.
  • Refine: a plan-only plan(io) that reads the marks' placed extent cells and writes pixel outputs (no domain, no train) — as fitText/grid-stack do.

Register it and expose an authoring factory:

registries.operators.register(myLayout.name, myLayout);   // into the Registries
export const myLayout = layoutFactory("myLayout");         // → myLayout("col").pick("x", "w")

Every column in output.add is .pick()-able (multi-output is free); supplying both a SQL and a JS impl makes it dual-substrate. The neighbouring registries extend the same way — a tick algo (registries.tickAlgos) earns an axis, a scheme (registries.schemes) a palette, a measurement provider (registries.measures) swaps the label-width heuristic for a real one. The full operator contract is in components/layouts/README.md.
