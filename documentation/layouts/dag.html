dag

dag places the nodes of a directed acyclic graph in topological layers — the layered (Sugiyama-style) layout used for dataflow/dependency diagrams — and emits an x, y per node, normalized to [pad, 1−pad] (the frame affine turns them into pixels). Like force it takes an edges table — dag('edges') — whose endpoint columns come from the declared join (edges.src → marks.id, edges.dst → marks.id) or an on override. There is no trained domain: positions carry no data-space meaning, so nothing is tickable.

Three deterministic steps (no simulation, no iteration):

  1. Layer — each node sits one layer below its deepest parent (longest path from the roots); roots on top, sinks at the bottom. A cycle's back-edge counts zero, so cyclic inputs terminate (with an arbitrary split inside the cycle).
  2. Order — one downward sweep sorts each layer by the barycenter (mean position) of its parents in the layer above, pulling connected nodes into vertical alignment and reducing edge crossings.
  3. Center — each layer's row centers on x = 0.5 with one uniform step shared across layers (the widest layer spans the full width).

This is the algorithm behind DVL's own compute-graph debugger: the playground's Graph tab is an ordinary DVL spec that reads the dataflow log (dvl_operator/dvl_op_io) back from the profiling database and draws it with dag (see design/2026-07-21-compute-graph-vis.html).

const v = canvas(640, 400);
v.dot('members', {
  ...dag('follows').pick('x', 'y'),
  r: 8, fill: ordinal('community'),
});
return v;

The members social graph as a hierarchy: dag('follows') reads the follows edge table (src/dst FK-joined to members.id) and stacks each member below the people they follow.

Options

dag(edges) takes the marks (the nodes, whose id column is the primary input) plus an aux edges relation named as the argument. Edges to nodes absent from a panel are ignored (a per-panel run sees only its panel's nodes). In a template it runs per cell by default — one layered graph per panel (perCell: true). js-only: an ordering algorithm over the whole node set has no meaningful SQL form.

Input Type Meaning
id (marks) int the node identity column, matched against edge endpoints
edges table the edge relation; src/dst (or the declared refs) are its endpoint columns

.pick() outputs:

Output Space Meaning
x relative coord x the node's slot in its layer's row, rows centered on 0.5
y relative coord y the node's layer: roots at the top, deepest layer at the bottom

Bind by spreading: ...dag('follows').pick('x', 'y'). An empty edge set puts every node in one top row; a single layer (or single node) centers at 0.5.

Examples

Nodes and edges — draw the links underneath with route(), which owns both endpoint borrows and clips each end to the node it lands on, so an edge meets the dot boundary nearest the far end rather than the center (the same pattern as force):

const v = canvas(640, 400);
const nodes = v.dot('members', { ...dag('follows').pick('x', 'y'), r: 8, fill: ordinal('community') });
v.link('follows', {
  ...route({
    src: nodes.via(eq('follows.src', 'members.id')),
    dst: nodes.via(eq('follows.dst', 'members.id')),
  }).pick('x1', 'y1', 'x2', 'y2'),
  stroke: '#b6bdc9',
});
return v;

Labelled nodes — a text layer rides the same dag placement (the layout is deterministic, so a second dag('follows') over the same nodes reproduces exactly the same positions):

const v = canvas(700, 400);
const nodes = v.dot('members', { ...dag('follows').pick('x', 'y'), r: 8, fill: ordinal('community') });
v.text('members', { ...dag('follows').pick('x', 'y'), text: field('name'), dy: -12, fill: '#343a40' });
v.link('follows', {
  ...route({
    src: nodes.via(eq('follows.src', 'members.id')),
    dst: nodes.via(eq('follows.dst', 'members.id')),
  }).pick('x1', 'y1', 'x2', 'y2'),
  stroke: '#b6bdc9',
});
return v;
