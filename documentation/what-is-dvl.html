What is DVL?

DVL (the Database Visualization Language) is a declarative visualization language built on one premise: a visualization is a database query. Not "uses a query to fetch its data" — the visualization itself, placement and all, is a query the engine compiles and the database executes.

In grammar-of-graphics terms, DVL looks familiar: you make plots from marks (dots, bars, lines, text) whose channels (x, y, fill, …) are bound to data columns through scales and layouts. The differences are underneath:

  1. Everything is a relation. A plot's marks are rows of a mark table derived from your source table. A scale's trained domain is a (tiny) relation. The tick marks on an axis are a relation. Even the diagram's diagnostics are a relation. Because everything is a relation, one algebra — and one optimizer — covers the whole pipeline.

  2. Scales and layouts are the same kind of thing: operators. An operator may train (reduce the data to a fit — a min/max envelope, a category index) and must apply (augment each mark row with output columns). A linear scale, a stacked-bar layout, and a force-directed graph layout are all operators; they differ only in what they train and what they add.

  3. Placement is normalized, then solved. Operators emit positions in a normalized [0, 1] frame — they never know pixels. A constraint solver arranges the boxes (panels, axes, gutters, margins), and a single affine step maps normalized positions into solved boxes. This split is why the same spec can facet, nest, and resize without any operator caring.

  4. It compiles to SQL. Mark tables, scale training, layout application, even the affine are SQL over DuckDB wherever possible, with JavaScript as a second execution substrate for algorithms that want it (force simulation, treemaps). The optimizer chooses where each step runs and inserts the conversions. A million rows scale-train in the database; only drawable pixels come out.

A complete example

import { canvas, linear, ordinal } from "$lib/dvl/api/builder";
import { renderSpec } from "$lib/dvl/executor/compile-spec";

const v = canvas(640, 480);
v.dot("penguins", {
  x: linear("bill_length_mm"),
  y: linear("body_mass_g"),
  fill: ordinal("species"),
})
 .axis("x")
 .axis("y");

const svg = await renderSpec(v.spec, v.registries, "", db);

Reading it the DVL way: dot declares a mark table over penguins; linear("bill_length_mm") binds the x channel through a scale operator that will train a {lo, hi} domain in SQL and apply (v − lo) / (hi − lo) to every row; ordinal("species") trains a category index and applies a palette lookup; .axis() asks for axes — which are themselves ordinary plots (a rule, ticks, and labels over the scale's tick relation), not a special renderer feature. renderSpec compiles all of that into a dataflow of SQL and JS steps, solves the frame, and reads back SVG.

How DVL relates to other tools

ggplot2 / Vega-Lite / Observable Plot DVL
Marks + channels + scales yes yes — the authoring surface is deliberately close
Where data lives in memory, in the tool in the database, until pixels
Layout algorithms built-in, special-cased operators in the same algebra as scales
Axes, legends renderer features ordinary plots over tick relations
Placement per-tool layout engine normalized frame + constraint solver
Extensibility plugins per concept one operator contract for scales and layouts

If you know Observable Plot, the authoring model will feel close; see Getting started. If you want the engine's internal story — the pipeline, the type system, execution — see Architecture.
