Getting started

DVL is authored in JavaScript/TypeScript against a fluent builder, and renders by compiling your spec to SQL over DuckDB. This page walks from an empty canvas to faceted, multi-layer plots. (For why DVL works the way it does, read What is DVL? first.)

Every block like the one below is live: it runs the current DVL build in your browser (DuckDB-Wasm), against the demo tables (penguins, members/follows, the customers/orders/products star). The sandbox matches the playground exactly — canvas, db, the scales/layouts, eq/on/fk/path, expr, px/field, and the aggregate helpers.

// bill length vs body mass, coloured by species — with axes and a legend.
const v = canvas(640, 320);
v.dot('penguins', {
  x: linear('bill_length_mm'),
  y: linear('body_mass_g'),
  fill: ordinal('species'),
}).axis('x').axis('y').legend('fill');
return v;

Set up a database

DVL draws from a Database, which wraps a DuckDB connection behind a backend adapter. In Node (tests, scripts):

import { Database } from "$lib/dvl/data/database";
import { DuckDBBackend } from "$lib/dvl/data/duckdb-backend";
import { createNodeDuckDB } from "$lib/dvl/data/duckdb-node";

const duck = await createNodeDuckDB();
const db = new Database({ backend: new DuckDBBackend(duck.conn) });

In the browser, use createBrowserDuckDB (DuckDB-Wasm) instead; everything else is identical.

Register data by rows, or point at tables that already exist in the database:

db.fromRows("penguins", [
  { bill_length_mm: 39.1, body_mass_g: 3750, species: "Adelie" },
  { bill_length_mm: 46.5, body_mass_g: 3500, species: "Chinstrap" },
  // …
]);
// also: db.table("existing_table"), db.fromQuery("view_name", "SELECT …")

Your first plot

import { canvas, linear } from "$lib/dvl/api/builder";
import { renderSpec } from "$lib/dvl/executor/compile-spec";

const v = canvas(640, 480);
v.dot("penguins", { x: linear("bill_length_mm"), y: linear("body_mass_g") })
 .axis("x")
 .axis("y");

const svg = await renderSpec(v.spec, v.registries, "", db);

canvas(width, height) starts a spec. Every mark method — dot, rect, bar, line, area, text, link, arrow, … — takes a source (a table name or Table handle) and a channels object. renderSpec compiles, executes, and returns an SVG string.

Two things to know about channels from the start:

  • A channel value can be a literal (fill: "#4e79a7", r: 6), a scale binding (x: linear("col")), or a layout output (next section). Positional literals are pixels.
  • Bare strings are literals everywhere, including the text channel — to show a column's values, bind it explicitly: text: col("label") (with col from $lib/dvl/ir/expr).

renderSpec mutates the spec as it compiles. Build a fresh spec per render (wrap authoring in a function) rather than rendering one spec twice.

Color and other output channels

Scales aren't only positional. ordinal maps categories to a palette; color maps a continuous value onto a ramp:

v.dot("penguins", {
  x: linear("bill_length_mm"),
  y: linear("body_mass_g"),
  fill: ordinal("species"),     // categorical → palette
  // fill: color("body_mass_g") // continuous → ramp
});

Pick by the column's nature, not the channel: ordinal trains SELECT DISTINCT, color trains min/max — a string column under color() is an error. See scales.

Bars: layouts and .pick()

A bar needs two numbers per axis — position and extent. Multi-output layouts produce them, and .pick() spreads their outputs onto channels:

import { bandX, barY, ordinal } from "$lib/dvl/api/builder";

v.bar("by_species", {
  ...bandX("species", { paddingInner: 0.2 }).pick("x", "w"),  // category → x + width
  ...barY("mean_mass").pick("y", "h"),                        // value → top + height
  fill: ordinal("species"),
});

bandX is D3's band scale as a layout (options: paddingInner, paddingOuter, padding, align); barY draws from a zero baseline. Each 1-D layout comes as an X/Y pair — swap to bandY + barX for a horizontal bar chart (same for stackX/stackY, pitchX/pitchY, eqX/eqY). Stacking is one substitution away:

v.rect("stacks", {
  ...bandX("species", { paddingInner: 0.2 }).pick("x", "w"),
  ...stackY({ weight: "n", group: "species", order: "island" }).pick("y", "h"),
  fill: ordinal("island"),
});

.pick() also renames — .pick({ x1: "x", y1: "y" }) maps a layout's x output onto the x1 channel. See layouts for the full set, including treemap, force, and fitText.

Lines and shared scales

Reusing one scale object binds several layers to the same trained domain:

const x = linear("bill_length_mm");
const y = linear("body_mass_g");
v.dot("penguins",   { x, y, fill: "#cfd3dc" });
v.dot("highlights", { x, y, fill: "#e15759" });   // same x/y domains as the layer above

Object identity is the authored "these are the same quantity" — distinct linear("col") calls train independently. For multi-series lines, line aggregates rows per series:

v.line("series", { x: linear("t"), y: linear("value"), stroke: ordinal("group") });

Facets and nesting

A container mark plus a join predicate turns any plot into small multiples. eqX splits the x-range equally among a table's rows; eq declares the parent↔child join:

import { eqX } from "$lib/dvl/api/builder";
import { eq } from "$lib/dvl/ir/refs";

v.rect("species", { ...eqX("species").pick("x", "w"), y: px(12), h: 236, fill: "#f5f6fa" })
 .dot("penguins",
      { x: linear("bill_length_mm"), y: linear("body_mass_g") },
      { on: eq("species", "species") });     // each panel shows its own species' rows

Scales are shared across panels by default; opt out per scale with linear("col", { shared: false }) for per-panel domains. .nest() names the pattern when a panel hosts several layers:

const panel = v.rect("clusters", { ...eqX("cluster").pick("x", "w"), y: px(12), h: 330 })
  .nest("fnodes", { on: eq("cluster", "cluster") });
const dots = panel.dot({ ...force("fedges").pick("x", "y"), r: 6 });
panel.link("fedges", {                        // edges layer in the same panels — route()
  ...route({ src: dots.via(eq("fedges.src", "fnodes.id")),   // attaches each end to a node
             dst: dots.via(eq("fedges.dst", "fnodes.id")) }).pick("x1", "y1", "x2", "y2"),
});

For ad-hoc side-by-side plots without a driving table, see subplots: v.subplot() / v.subplots({...eqX("i").pick("x","w")}).

Connecting marks across plots: via

Layers can borrow another plot's solved positions through a join — how node-link diagrams, annotations, and connectors are built:

const dots = v.dot("penguins", { x: linear("bill_length_mm"), y: linear("body_mass_g") });
v.link("highlights", {
  ...dots.via(eq("highlights.species", "penguins.species")).pick({ x1: "x", y1: "y" }),
  x2: linear("bill_length_mm"), y2: linear("body_mass_g"),
  stroke: "#e15759",
});

via() accepts multiple predicates for composite keys, and works into nested plots. Borrowed positions arrive in pixels and are not re-transformed. For node-link edges, wrap the two endpoint vias in route({ src, dst }) — that is how an edge attaches to each node's boundary (the circle or rect edge) instead of borrowing its raw center. See cross-mark references and via → route().

Rendering

renderSpec returns SVG. In the browser, insert it; in Node, rasterize with render/png (@resvg/resvg-js) or write the SVG to disk. To watch the engine work, pass a debug sink — every compile pass, SQL statement, and solver iteration lands in queryable dvl_* tables, which DVL can then visualize with itself:

import { DuckDbSink } from "$lib/dvl/debug/duckdb-sink";
const sink = new DuckDbSink(db.backend, 2);
const svg = await renderSpec(v.spec, v.registries, "", db, {}, sink);
// then: spanTimeline(db, sink.runId), sqlProfile(db, sink.runId), …

See debugging.
