DVL Documentation

DVL is a visualization language that treats a chart as a database query. You declare marks over relations, bind their channels to scales and layouts, and DVL compiles the whole visualization — placement included — into SQL over DuckDB plus a small solved constraint system. Data never leaves the database until there are pixels to draw.

const v = canvas(640, 480);
v.dot("penguins", { x: linear("bill_length_mm"), y: linear("body_mass_g"), fill: ordinal("species") })
 .axis("x")
 .axis("y");
const svg = await renderSpec(v.spec, v.registries, "", db);

These docs follow the shape of Observable Plot's documentation: a short getting started, concept pages under features, one reference page per instance under marks, layouts, and scales, and a worked example gallery. The full page tree is in the sidebar: SUMMARY.md.

Deferred decisions & roadmap

Choices made deliberately and work pushed to the future, so they read as intent rather than accident (full rationale in design/CELL_KEY_VIA.html §7–§8):

Decisions that stand until revisited

  • Inline JS reductions don't serialize — accepted. A via resolution's opaque function rides the plan object (the same precedent as embedded resolved operators); plans are session-local today. Registered by-name reductions come back only if plan serialization becomes real.
  • Unprovable join multiplicity passes. Only provably multi-match joins (non-eq, or eq that visibly misses a key) demand a resolve; joins the FK catalog can't judge compile as single-match. Making the judgment total needs a populated catalog plus a declareKey for non-FK unique keys, then a strict flag flips unknown → error.
  • Empty match set ⇒ NULL, uniformly — even for count (LEFT-JOIN semantics; "no match" is a value, never a dropped mark).
  • The two force substrates are different simulations. Same contract — normalized positions, deterministic per substrate — different exact coordinates. A test that pins force pixels is pinning a substrate choice.
  • Borrow shadowing: the referent wins. A via output that collides with an existing mark column replaces it (that's the point of the via), with a one-time warning; a declared replace stays silent.
  • Computed resolutions must be named. { resolve: { spend: { fn, arg } } }, picked by 'spend' — an expression/JS reduction mints a quantity no relation has a column for, so the map key is its declaration; the unnamed form is rejected.

Deferred work

  • required-columns pass — proving join-key columns survive to the hop that needs them. Fires on nothing today (augment-only pipelines never drop columns); becomes real with the first projecting relop.
  • Child-extent aggregation — host extents as a grouped reduce over children's placed marks (by: dvl_cell → an extents table). The standalone reduce op is its prerequisite and is built; the layout wiring is not.
  • Intermediate hop columns in JS match sets — an opaque reduction's matches carry only the referent's rows; hop columns are reachable through m in expression args but are not flattened into the match rows (needs a collision rule).
  • Level-3 cell inference — auto-defaulting a mark's panel from its position vias. Kept explicit (pick({ cell: 'dvl_cell' })) deliberately; inference would hide the frame decision.
  • Nested subplots as database partitions — LANDED (docs/2026-07-15-per-cell-layouts.html §7, executed per §7e): a template cell scopes the WHOLE database — a nested layer's partition σ derives from the FK catalog (conjoin default; { scope: 'union' } for local-subgraph multiples), cross-panel rows don't exist at the data level, "all edges" is a top-level layer with { frame: srcVia }, dropped rows are reported as data (sigmaDropped), and the cell pick is deprecated. Remaining from that design: withdrawing the deprecated pick, and σ for AUX inputs (force's edges by name resolution — blocked on cell-backed aux bindings), which would delete the d3 impl's hand-rolled dangling-link filter. The logic, spelled out (another paper-worthy instance of semantics doing the deciding): a nest key is a join key, and a join key IS a partition — so take the partition seriously and it partitions not one table but the database, since every other relation reaches the host through declared FKs. Then (i) "does the cross-panel edge render?" stops being a rendering flag and becomes name resolution — inside a panel the edge simply isn't in scope; (ii) the one genuine ambiguity relocates to where it belongs, the data level: how σ extends at a relation with SEVERAL FK paths to the host (conjoin = strict partition, union = local-subgraph multiples) — declared once about the schema, never per layer; (iii) wanting global data inside a panel becomes an explicit opt-out (global data + frame assignment), not a laxness of nesting; and (iv) data, fits (scope-of-fits above), and frames end up scoped by ONE environment — the panel. Same shape as the Space and fit stories: classify the thing (how does this relation reach the host?), and behavior falls out of the classification instead of a per-use option. Worth writing up alongside them.
  • Joins inside scalar expressions — a cross-table ref nested in an arithmetic expression still throws; bind the via output to its own channel instead.
  • Space checking of resolution arguments — a computed output's declared space is trusted, not yet verified against the argument expression's inferred space.
  • e2e-duckdb assertion drift — a set of pre-existing margin/gutter mismatches in the DuckDB e2e suite predates the current work and needs its own reconciliation pass.
  • PlotParams.zIndex is declared but unread — the renderer draws in authoring order; a layer that must sit under an earlier one (edge lines under node boxes) can't say so yet.
  • perCell → scope-of-fits — the per-cell layout default landed as an explicit operator bit; the cleaner mechanism (scope as a per-FIT property derived from the declaration shape — per-row work is scope-invariant, factored domains default shared, whole-set execs default cell; gather-scatter and keyed execution as the coarsen/confine coercions on one axis) is designed in docs/2026-07-15-per-cell-layouts.html §6 and awaits the exec-operator audit. Why the fit is the right home for scope: scope is about computing properties of the data — metadata (a domain, a normalization, a box solve) — and fits are exactly where the engine represents metadata; per-row values have no scope because they aren't metadata. This is the same move as the rest of the design — the type system and its semantics doing the deciding (cf. "the type system enumerates the cases"): classify the value (row datum vs fit), and the legal/default scoping falls out of the classification rather than out of a flag.
