Features

One page per concept. Marks, layouts, and scales each have a per-instance reference (marks/, layouts/, scales/); the pages here explain the mechanisms those instances plug into — how a spec becomes pixels.

Feature What it covers
Plots and marks a plot = one mark over one relation; the canvas hosts plots and solves their shared frame; layering, containers, params
Channels the channel-value language: literals, scale bindings, .pick() outputs, field()/cexpr expressions, arithmetic with lifted borrows
Scales domain training and application; sharing by object identity; shared: false; treatments (nice, zero, explicit); color as an output space
Layouts arrangement operators in three classes — produce, refine, measure — multi-output .pick(), and the X/Y pair convention
The frame normalized → pixel: the constraint solver, the affine, axes as ordinary plots, axis options (titles default ON), margins and insets
Join paths ONE Path vocabulary (eq/on/fk/path, bare-string shorthand) for every join site; resolution against the FK catalog; multiplicity judgment
Facets and nesting small multiples = a container plot + a nest key; cell borrows; the partition model (scope, frame: owner semantics)
Via borrowing across relations: plot vs table referents, resolutions over multi-match joins, route() boundary attachment
Subplots ad-hoc side-by-side panels over a virtual panels table — v.subplot() / v.subplots({...})
Data the Database front door: loading, branches, modeling operators, the unit table (UNIT), the FK catalog, substrates
Debugging the DuckDbSink record of every pass, operator, SQL statement, and solver iteration — explored with SQL or DVL's self-visualizations

Reading order: if you are new, Getting started walks the common path; these pages go deeper one mechanism at a time. Architecture is the whole-engine view the mechanisms compose into.
