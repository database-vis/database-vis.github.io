Plots and marks

A plot is one mark type over one source relation; a canvas hosts plots and solves their shared frame. Marks are deliberately simple — a channel list and a draw function — so all intelligence about position lives in scales and layouts. Covers: canvas, the mark methods, layering order, containers (rect/bar/cell can host nested plots), and params.

// three layers over one canvas share a frame: the cloud, a per-species highlight
// (modelled inline on the db branch), and labels riding the highlight's scales.
const heaviest = db.table("penguins").top(1, { col: "body_mass_g", desc: true }, { partition: ["species"] });
const v = canvas(640, 360);
const x = linear("bill_length_mm"), y = linear("body_mass_g");
v.dot("penguins", { x, y, fill: "#cfd3dc" });
v.dot(heaviest, { x, y, r: 5, fill: "#e15759" });
v.text(heaviest, { x, y, text: field("species"), dx: 8, anchor: "start", fontSize: 10, fill: "#333" });
return v;

Channels

This is the concept outline; the full channel-value language — every expression kind, the data/pixel and resolution model, and a reference table — has its own page.

How channel values bind: literals (positional literals are pixels), scale bindings, layout outputs via .pick(), expressions (col() from ir/expr), and cross-mark references. The bare-string rule (strings are literals everywhere — including text), channel kinds (positional-x/y, pixel, color, shape, text), and how a mark's channel kinds type-check what you bind. Arithmetic channels: an expression may combine columns and literals — x: binary("+", col("total"), literal(5)) — and compiles to both substrates (SQL + JS) through the one ScalarExpr lowering; a via/aggregate nested inside arithmetic is lifted into its own join automatically, so y: borrowed + 5 works (a pixel offset on a placed value, a data offset on a borrowed column).

// literal vs scale vs column: x/y are SCALE bindings; r: 4 and fill: '#4e79a7' are
// LITERALS (a bare string is never a column read); the label reads a column with field().
const v = canvas(640, 320);
const x = linear("bill_length_mm"), y = linear("body_mass_g");
v.dot("penguins", { x, y, r: 4, fill: "#4e79a7", fillOpacity: 0.6 });
v.text("penguins", { x, y, text: field("island"), dy: -8, fontSize: 8, fill: "#888" });
return v;

// ARITHMETIC channels: each arrow's head is a data expression — cexpr`…` builds the tree (bare
// names are columns), compiling once to both substrates (SQL + JS) via the ScalarExpr lowering.
// Feeding it through the SAME scale as the tail (x()/y()) puts both endpoints in one domain — one coupled axis.
const flow = db.derive("flow", `
  SELECT bill_length_mm AS x, body_mass_g AS y,
         (bill_depth_mm - 17) * 0.3 AS dx, (flipper_length_mm - 200) * 40 AS dy
  FROM penguins`);
const v = canvas(640, 360);
const x = linear(), y = linear();
v.arrow(flow, {
  x1: x("x"), y1: y("y"),
  x2: x(cexpr`x + dx`),
  y2: y(cexpr`y + dy`),
  headType: "barb", headSize: 5, stroke: "#4e79a7", strokeOpacity: 0.6,
}).axis("x").axis("y");
return v;

Scales

Operators that train a domain from data and apply a normalization per row. Continuous (linear, log, sqrt, pow, symlog), categorical (ordinal, point), and binned (quantize, threshold) families; on a color channel any continuous scale is a sequential ramp and ordinal is a categorical palette (a scheme names the palette) — color is an output space, not a scale family. Domain sharing (object identity, facet defaults, shared: false); domain treatments (nice, zero, explicit). Reference: scales/.

// ONE scale object bound across two relations declares "the same quantity": the domains
// UNION and both layers share a frame — the species means land exactly on the cloud.
const bySpecies = db.table("penguins").groupby(["species"], { bill: avg("bill_length_mm"), mass: avg("body_mass_g") });
const v = canvas(640, 360);
const x = linear(), y = linear();
v.dot("penguins", { x: x("bill_length_mm"), y: y("body_mass_g"), fill: "#cfd3dc" });
v.dot(bySpecies, { x: x("bill"), y: y("mass"), r: 6, fill: ordinal("species") })
 .axis("x").axis("y").legend("fill");
return v;

Layouts

Operators that compute arrangements — usually multi-output (.pick("x", "w")). The three classes: produce (data → normalized positions: bandX, barY, stackY, eqX, force, treemap, proportional*), refine (positions → positions, post-placement: fitText, grid-stack), measure (positions → sizes, feeding the demand loop). Reference: layouts/.

Each 1-D producer is an X/Y pair — bandX/bandY, barX/barY, stackX/stackY, pitchX/pitchY, eqX/eqY — so a vertical bar (bandX + barY) mirrors to a horizontal one (bandY + barX) by swapping the axis; eqX splits columns, eqY rows.

// multi-output layouts spread into channels with .pick(): bandX gives {x, w}, stackY {y, h}.
const stacks = db.table("penguins").groupby(["species", "island"], { n: count() });
const v = canvas(640, 340);
v.rect(stacks, {
  ...bandX("species", { paddingInner: 0.2 }).pick("x", "w"),
  ...stackY({ weight: "n", group: "species", order: "island" }).pick("y", "h"),
  fill: ordinal("island"),
}).axis("x").legend("fill");
return v;

The frame

Where pixels come from: operators emit normalized [0, 1] coordinates; a constraint solver arranges boxes (canvas, panels, axis gutters, margins); the affine maps normalized → pixel per plot and axis. Axes are ordinary plots over a scale's tick relation — placement, merging of duplicate sibling axes, tick-count-from-extent, and label measurement all flow through the same dataflow as your marks. Covers .axis(), axis options, margins/insets, and pixel-passthrough channels for precomputed geometry.

// axes are ordinary plots over the scale's tick relation: pin a tick count on x, place
// the y axis on the RIGHT — the gutters are solved from measured labels, not guessed.
const v = canvas(640, 320);
v.dot("penguins", { x: linear("flipper_length_mm"), y: linear("body_mass_g"), fill: "#4e79a7" })
 .axis("x", { ticks: { count: 4 } })
 .axis("y", { placement: "right" });
return v;

Axis options — pass them to the .axis(dim, opts) directive or scale-level as linear("c", { axis: {…} }) (the directive wins on conflict; axis: false suppresses the axis):

Option Type Default Meaning
title boolean | string | object none axis title — true derives the text from the scale's column, a string sets it, an object styles the text mark (text, fontSize 11, fill/color #333, opacity, dx/dy nudge, rotate, anchor, baseline)
ticks number | { count, algorithm } auto tick count, or a count + a registered tick algorithm
tickLength number 5 tick-mark length; label nudges and the gutter pad derive from it
fontSize number 10 tick-label font size
placement "top"|"right"|"bottom"|"left" auto pin the axis to a side (the .axis() directive; scale-level uses side)
place "inside" | "outside" role rank order among co-located bands on the same side
gap number 0 px before the band along the stack normal
pad number | per-side 0 inner padding between the band edges and its ticks/labels 
// axis TITLES: `true` derives the title from the column; a string sets it explicitly.
const v = canvas(640, 340);
v.dot("penguins", { x: linear("flipper_length_mm"), y: linear("body_mass_g"), fill: "#4e79a7" })
 .axis("x", { title: true })
 .axis("y", { title: "body mass (g)" });
return v;

Join paths

"Relate this relation to that one, on these columns, possibly through intermediates" is one concept — a Path — and every join site in DVL consumes it: nest keys, cross-mark and cross-table via, layout aux inputs (force's edges). One constructor vocabulary:

eq('penguins.island', 'islands.island')   // an explicit predicate (also lt/le/gt/ge/neq)
on('island')                              // same-column eq shorthand
fk('islands')                             // a named FK hop — expands against the catalog
path('friends', 'penguins', 'islands')    // a multi-hop chain of FK hops
// …and OMITTING the path infers the unique FK between source and target

Paths resolve once (against the database's FK catalog — see Data); what varies per site is policy, not type: a nest key must resolve to all-eq conditions (it feeds a partition key), while a via admits any predicate — a range join just matches a set, which is what resolutions are for. The resolver also judges multiplicity: a hop that lands on a known key (an FK's target) is provably single-match; a non-eq hop is provably multi; multi without a declared resolution is a build-time error, never a silent fan-out.

// a node-link diagram: force places the members; route() OWNS both endpoint borrows and
// clips each edge to the node BOUNDARY nearest the far end. The `eq('follows.src','members.id')`
// predicates ARE the join paths — `on('col')` (same-column eq), `fk('table')` (named hop), and
// `path(...)` (multi-hop) are the shorthands the section lists, resolved against the same FK catalog.
const v = canvas(640, 420);
const nodes = v.dot("members", { ...force("follows").pick("x", "y"), r: 8, fill: ordinal("community") });
v.link("follows", {
  ...route({ src: nodes.via(eq("follows.src", "members.id")),
             dst: nodes.via(eq("follows.dst", "members.id")) }).pick("x1", "y1", "x2", "y2"),
  stroke: "#bbb", strokeOpacity: 0.6,
});
return v;

Because route() owns the endpoint joins, each edge lands on the node's boundary (a dot clips on its circle) — not its center — so arrowheads and links meet the mark cleanly.

Facets and nesting

Small multiples are a container plot plus a nest key — any join path:

panel.nest('penguins', { on: on('island') })       // same-column eq shorthand
panel.nest('penguins', { on: fk('islands') })      // a named FK hop
panel.nest('penguins')                             // infer the unique FK from the catalog
panel.link('friends', { on: path('friends', 'penguins', 'islands') })  // MULTI-HOP: the link
// needn't carry a denormalized panel column — the key travels friends.src → penguins → islands

A multi-hop key compiles to one join per hop; the panel key (dvl_cell) threads through scales, ticks, and placement. Better still, a layer whose referent is already nested can borrow the referent's cell instead of re-deriving it — pick the reserved cell output:

panel.link('friends', {
  ...np.via(eq('friends.src', 'penguins.id')).pick({ x1: 'x', y1: 'y', cell: 'dvl_cell' }),
  ...np.via(eq('friends.dst', 'penguins.id')).pick({ x2: 'x', y2: 'y' }),
})

One declared FK, read twice: the cell resolves early (it stands in for the nest key, before layout), the positions resolve late (post-placement pixels). The borrow path must be provably single-match end to end — declare the FKs it walks. Also covers .nest() multi-layer panels and per-panel vs shared domains (shared: false). route() composes with these borrows too: wrap the two position vias in route({ src, dst }) to clip each edge to the nested node's boundary while the cell borrow still resolves the panel.

Which rows belong to a panel? Take the partition seriously: a nested subplot scopes the whole database — inside a panel, a table name denotes its restriction to that panel, derived through the declared FK paths. A nested layer over an edge table needs no on and no borrow: panel.link('follows', { …position vias… }) derives its partition, and a cross-panel edge doesn't get filtered — it simply doesn't exist in either panel's data (the strict, conjoin-the-paths default). { scope: 'union' } puts a multi-path row in every panel it touches (local-subgraph small multiples). A layer that genuinely wants global data with a per-panel identity is a top-level layer with a frame: opt naming its assigning path (v.link('follows', …, { frame: srcVia })) — an explicit opt-out of the partition. The rows a conjoin drops are reported as data (the layer's sigmaDropped diagnostics relation), and the old pick({ cell: 'dvl_cell' }) spelling still works with a deprecation warning.

// a container rect per species + a nest key (any join path): panels partition the data,
// and the shared x/y scales keep every panel comparable.
const species = db.table("penguins").groupby(["species"], {});
const v = canvas(760, 280);
v.rect(species, { ...eqX("species").pick("x", "w"), y: px(12), h: 236, fill: "#f5f6fa" })
 .dot("penguins", { x: linear("bill_length_mm"), y: linear("body_mass_g"), fill: ordinal("species") },
      { on: eq("species", "species") });
return v;

Cross-mark and cross-table references

This section keeps the resolution grammar reference; the full mechanism — both referent kinds, top-level vs nested consumers, frame assignments, timing, and the placeholder machinery — has its own page.

One verb, two referent kinds — the receiver says which, and the type system carries the difference (same mechanism, different Space):

// PLOT referent: borrow SOLVED positions — results arrive in PIXELS, never re-placed
np.via(eq('friends.src', 'penguins.id')).pick({ x1: 'x', y1: 'y' })

// TABLE referent: borrow DATA values — results arrive in DATA space, free to feed a scale
db.table('penguins').via(eq('friends.src', 'penguins.id')).pick({ w: 'mass' })

Composite keys, multi-hop paths, and nested referents are supported. A join that can match several referent rows per mark must declare its resolution — how the match set reduces to one value:

// a bare AggFn applies to EVERY picked channel independently (centroid of my cities)
cityDots.via(eq('regions.id', 'cities.region'), { resolve: 'avg' }).pick({ cx: 'x', cy: 'y' })

// positional forms order within the match set (each order's LAST quote before it)
quoteDots.via(gt('orders.ts', 'quotes.ts'), { resolve: { fn: 'last', order: 'ts' } })
         .pick({ fill: 'price' })

// an EXPRESSION argument over the two-sided row — m is your mark, f is one match;
// a computed resolution declares a NEW output name (the map key) and is picked by it
quoteDots.via(gt('orders.ts', 'quotes.ts'),
              { resolve: { spend: { fn: 'sum', arg: expr`f.price * f.qty` } } }).pick('spend')

// an opaque JS reduce — a bare function as the map value: (matches, mark) => value
cityDots.via(eq('regions.id', 'cities.region'), {
  resolve: { far: (matches, mark) =>
    Math.max(...matches.map(c => Math.hypot(c.x - mark.x, c.y - mark.y))) },
}).pick({ r: 'far' })

// a NAMED reducer — a string map value looks up a registered JS reducer by name
// (register with registries.reducers; keeps the spec serializable, unlike an inline fn)
cityDots.via(eq('regions.id', 'cities.region'), { resolve: { pick: 'mode' } }).pick({ fill: 'pick' })

Declarative resolutions lower entirely to SQL (a staged join, one GROUP BY dvl_mark reduce, a LEFT-JOIN attach — a mark with no matches gets NULL, never dropped); a JS reduce (inline function or named registered reducer) materializes its match sets. The expr tagged template writes arguments as plain strings — m.<col>/f.<col> reference the two sides. A computed output's space defaults to data; declare one with { fn, arg, space: pixelSpace(len('x')) } when the value is really a pixel quantity. If a borrowed name collides with an existing mark column, the referent's value wins (that's the point of the via) — with a one-time warning. A provably multi-match join without a resolution is a build error; the lenient single-match default for unprovable joins can be tightened by passing { strictMultiplicity: true } to renderSpec/compileSpec, which makes every unprovable multiplicity a build error too. The basis for node-link diagrams, annotations, connectors, and ER-style diagrams.

// PLOT referent + a resolution, live: each species label sits at the CENTROID of its
// dots — the via matches MANY dots per species; resolve: 'avg' reduces every picked
// channel independently. The borrowed values are solved pixels, never re-placed.
const species = db.table("penguins").groupby(["species"], {});
const v = canvas(640, 360);
const dots = v.dot("penguins", { x: linear("bill_length_mm"), y: linear("body_mass_g"), fill: ordinal("species"), fillOpacity: 0.5 });
v.text(species, {
  ...dots.via(eq("species", "species"), { resolve: "avg" }).pick({ x: "x", y: "y" }),
  text: field("species"), fontSize: 12, fill: "#333", anchor: "middle",
});
return v;

Subplots

v.subplot() appends a row to a virtual panels table and hosts a child plot in it; v.subplots({ ...eqX("i").pick("x", "w") }) chooses the panels' layout exactly the way any plot chooses one. Auto row/column when no layout is bound.

// two panels over the panel-index virtual table; both encode body_mass on y, so the
// solver UNIONS that scale and the interior y-axis merges away — ONE axis speaks for both.
const v = canvas(720, 300);
v.subplot().dot("penguins", { x: linear("bill_length_mm"), y: linear("body_mass_g") });
v.subplot().dot("penguins", { x: linear("flipper_length_mm"), y: linear("body_mass_g") });
v.subplots({ ...eqX("i").pick("x", "w") });
return v;

Data

This is the concept outline; the full data-layer reference — branches, loading data, the Table modeling operators, the raw-SQL and expr/cexpr string templates, and the FK catalog — has its own page.

The Database wrapper: fromRows, fromColumnar, table, fromQuery; backends (DuckDB-Wasm in the browser, the same engine in Node); schema introspection; when data crosses between SQL and JS and what it costs.

The FK catalog is what join paths resolve against — it powers fk() hops, inferred nest keys, and the multiplicity judgment (FK targets are keys, so landing on one proves single-match):

const db = new Database({ backend });
db.declareFK({ fromTable: 'friends', fromCols: ['src'], toTable: 'penguins', toCols: ['id'] });
await db.loadFKs();                     // or introspect the backend's declared FKs
const v = canvas(640, 480, { db });     // hand the catalog to the builder/resolver

Without a catalog, explicit predicates still work everywhere; only inference and provability degrade (unprovable joins compile as single-match — today's lenient default).

// the Database wrapper in the small: `db.fromRows` builds a relation from literal rows,
// and the canvas queries it like any base table — the front door the section opens with
// (fromRows / fromColumnar / table / fromQuery). The FK catalog above is what join paths
// then resolve against.
db.fromRows("quarters", [
  { q: "Q1", rev: 120 }, { q: "Q2", rev: 180 }, { q: "Q3", rev: 90 }, { q: "Q4", rev: 210 },
]);
const v = canvas(420, 300);
v.bar("quarters", {
  ...bandX("q").pick("x", "w"),
  ...barY("rev", { zero: true }).pick("y", "h"),
  fill: "#4e79a7",
}).axis("x").axis("y");
return v;

Debugging

Pass a DuckDbSink to renderSpec and every compile pass, operator run, SQL statement, cell write, and solver iteration is recorded into dvl_* tables — then explore them with SQL, or with DVL's own self-visualizations (spanTimeline, sqlProfile, convergenceTrace, convergenceResidual, entityCounts, erDiagram).
