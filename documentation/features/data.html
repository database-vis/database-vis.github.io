The data layer

Every DVL visualization is a query over a database. The Database object is the front door: it registers your data, hands out Table handles, models new relations from old ones, resolves foreign keys, and owns the DuckDB backend (DuckDB-Wasm in the browser, the same engine in Node). This page is the reference for that layer — what a developer touches before the first canvas(...).

Two ideas run through all of it:

  • Everything is a relation. A base table, a groupby, a raw-SQL view — all are Tables the canvas can plot. There is one query language underneath, and marks never care which kind of relation they read.
  • Modeling is metadata, not materialization. A modeling step (groupby, filter, join, …) records a query object — a fragment of SQL whose inputs point at its dependencies — and never writes a view. At render time the whole chain from a modeled table back to the base tables folds into one query; only base tables actually hit the backend. So you can reshape freely without a view lifecycle to manage.
import { Database } from "$lib/dvl/data/database";
const db = new Database({ backend });          // backend = a DuckDBBackend
// mode: "strict" throws on warnings; "non-strict" (default) routes them to onWarning.

In the live playground and the documentation examples, the sandbox hands you a db that is already booted with the demo base tables — penguins, members/follows, and a normalized sales star customers/orders/products — with their foreign keys declared. That db is a branch (see Branches); everything below works the same on it.

Loading data

Four ways in, each returning a Table synchronously — spec construction never blocks on the backend. The actual push into DuckDB is lazy: it happens on the first query that needs the relation (or when you call db.register() explicitly).

Method Source Schema
db.fromRows(name, rows) an array of JS objects inferred from the rows
db.fromColumnar(name, cols) a { col: [...] } columnar bundle inferred from the columns
db.table(name) a relation already in the backend introspected on demand
db.fromQuery(name, sql) opaque user SQL, registered as a view introspected on demand 
// fromRows builds a base relation from literal JS rows; the canvas queries it like any table.
db.fromRows("quarters", [
  { q: "Q1", rev: 120 }, { q: "Q2", rev: 180 }, { q: "Q3", rev: 90 }, { q: "Q4", rev: 210 },
]);
const v = canvas(440, 300);
v.bar("quarters", { ...bandX("q").pick("x", "w"), ...barY("rev", { zero: true }).pick("y", "h"), fill: "#4e79a7" })
 .axis("x").axis("y");
return v;

You can pass a base table to a mark by name (v.dot("penguins", …)) or by its Table handle (v.dot(db.table("penguins"), …)) — a name is resolved through the branch, so both reach the same relation. await db.register() forces all pending JS data into the backend up front (only needed if you query the DB directly before rendering); table.schema() resolves and caches a relation's columns.

Branches

A branch is a lightweight child overlay of a database: db.branch(name?). Names resolve through the branch — a relation defined on the branch shadows the parent, and a miss falls through to the parent, then to a base table of that name. This is how DVL isolates work: the documentation runs every example on its own throwaway branch so one example's modeled tables can never leak into the next, and the base data is never mutated.

const scenario = db.branch("q3-forecast");   // a child overlay; parent untouched
scenario.table("orders").filter("year = 2007");   // modeled ONLY on this branch
// … build and render against `scenario` …
await scenario.release();                    // reclaim anything it materialized

Because modeling records query objects rather than views, a branch needs no view lifecycle to be correct — release() only has to drop the handful of things that genuinely created backend objects (materialize, derive). A child holds its parent alive; reclamation cascades up as children release.

Two escape hatches when the fold isn't enough:

  • db.materialize(identity) — snapshot a modeled relation into a real backend table. Use it when a deep fragment stack is re-queried in a hot loop; the identity then resolves to the table (a base leaf) instead of re-folding. Namespaced, so release() reclaims it.
  • db.derive(identity, sql) — register a branch-scoped view from raw SQL, returning the bound Table. This is the hatch for shapes the structural operators don't cover (window→pixel geometry, exotic SQL). Its SQL stays visible in your code — no hidden boot view — and it is reclaimed on release like a materialized relation.
// derive is the raw-SQL escape hatch: a branch-scoped view whose SQL is right here. It joins the
// sales star and rolls revenue up to region — then the canvas plots it like any table.
const byRegion = db.derive("region_rev", `
  SELECT c.region, sum(o.qty * p.price) AS revenue
  FROM orders o
  JOIN customers c ON o.cust_id = c.id
  JOIN products  p ON o.product_id = p.id
  GROUP BY c.region`);
const v = canvas(440, 300);
v.bar(byRegion, { ...bandX("region").pick("x", "w"), ...barY("revenue", { zero: true }).pick("y", "h"), fill: ordinal("region") })
 .axis("x").axis("y").legend("fill");
return v;

Table and its operations

A Table is the uniform handle over any relation. Beyond schema(), it carries the modeling operators — each returns a new Table (a fresh identity on the branch) and leaves its input untouched, so operations chain. They are query objects: nothing runs until render, and the whole chain folds into one query.

Operator Signature Produces
copy copy(name?) a working copy under a fresh identity — reshape it without shadowing the original
filter filter(predicate, name?) same columns, fewer rows; predicate is a raw SQL boolean over the row ("species <> 'Gentoo'")
groupby groupby(keys, aggs, name?) one row per distinct keys combo + one column per named aggregate
rank rank(order, { partition?, as?, name? }) all rows + a 1-based rank column, per partition, by order
top top(n, order, { partition?, name? }) the top-n rows per partition by order (rank-then-filter)
join join(other, on, name?) this table's columns + other's; on is a shared column or { left, right }
rollup rollup(other, on, key, aggs, name?) join(other, on).groupby([key], aggs) in one call — "denormalize then summarize"
normalize normalize(col, { bin? }) splits col into its own keyed dimension table + rewrites this table to carry <col>_id (and declares the FK); returns the dimension
unpivot unpivot(cols, { into?, value?, name? }) stacks wide cols into long (name, value) rows (SQL UNPIVOT)
withKey withKey(as?, name?) adds a stable 1-based surrogate key column (the tuple identity a via joins on)

order for rank/top is a column name (ascending) or { col, desc }. Every operator takes an optional final name to pin the resulting relation's identity (otherwise a fresh, reproducible one is minted).

// operations CHAIN — each returns a new modeled table on the branch. Filter rows, then keep the
// heaviest two penguins per species; the base `penguins` (grey cloud) is untouched.
const heavy = db.table("penguins")
  .filter("body_mass_g > 3400")
  .top(2, { col: "body_mass_g", desc: true }, { partition: ["species"] });
const v = canvas(560, 320);
const x = linear("bill_length_mm"), y = linear("body_mass_g");
v.dot("penguins", { x, y, fill: "#d0d5dd" });
v.dot(heavy, { x, y, r: 6, fill: ordinal("species") }).axis("x").axis("y").legend("fill");
return v;

Aggregates for groupby/rollup come from a small vocabulary: count(), sum(arg), avg(arg), min(arg), max(arg), median(arg). An arg is a bare column name or a raw SQL expression { sql: "qty * price" }:

// rollup = join then aggregate: revenue per product category. `orders` carry qty; the join to
// `products` brings price; sum({ sql: 'qty * price' }) is a raw-SQL aggregate argument.
const byCat = db.table("orders")
  .rollup(db.table("products"), { left: "product_id", right: "id" }, "category",
          { revenue: sum({ sql: "qty * price" }) });
const v = canvas(440, 300);
v.bar(byCat, { ...bandX("category").pick("x", "w"), ...barY("revenue", { zero: true }).pick("y", "h"), fill: ordinal("category") })
 .axis("x").axis("y").legend("fill");
return v;

normalize is the modeling step that turns a denormalized column into a star: it mints a dimension table with a dense surrogate key, rewrites the fact table to carry the FK, and declares that FK — so via(fk(col)) and nest({ on: fk(col) }) traverse it with no extra wiring ({ bin: n } buckets a continuous column first). withKey adds the surrogate identity that parallel-coordinate and link diagrams join on.

Strings: raw SQL and the expression templates

DVL has two ways to write an expression as text, with different powers:

Raw SQL is the data layer's escape hatch — filter(predicate), derive(id, sql), fromQuery(name, sql), and the { sql: … } aggregate argument. It is exactly SQL, so it is maximally expressive but SQL-only: a relation defined by raw SQL stays in the database (it can't run in the JS substrate). Reach for it when the structural operators don't cover a shape.

Tagged templates build a small expression AST that lowers once to both substrates (SQL in the database, JS in the browser) — you never pick a backend:

  • expr`…` (from algebra/expr-template) — a scalar expression over the two-sided row a via resolution sees: bare m.<col> is the mark side, f.<col> the referent side. Operators + - * / with normal precedence and parentheses; unary floor, ln, abs, sqrt, nullif0; bare numbers become literals; ${…} splices another expression node. It is the argument form for computed via resolutions: { resolve: { spend: { fn: "sum", arg: exprf.price * f.qty } } }.
  • cexpr`…` (from ir/expr) — the channel expression template: bare names are columns, ${…} splices a channel expression. cexpr`x + dx` ≡ binary("+", field("x"), field("dx")). Full treatment on the Channels page.

The rule of thumb: tagged templates for arithmetic that should follow the data into either substrate; raw SQL for whole-relation shapes that live in the database.

The FK catalog

Foreign keys are what DVL's join paths resolve against — fk() hops, inferred nest keys, and the multiplicity judgment (landing on an FK's target column is provably single-match, so it never fans out silently).

db.declareFK({ fromTable: "orders", fromCols: ["cust_id"], toTable: "customers", toCols: ["id"] });
await db.loadFKs();                 // also pull in the backend's introspected FKs (deduped)

The catalog is branch-scoped with inheritance: db.fks is this branch's edges plus every ancestor's, so base-table FKs declared on the root are visible in every branch. normalize auto-declares the FK it creates. Without a catalog, explicit predicates still work everywhere — only inference and provability degrade (an unprovable join compiles as single-match, today's lenient default; tighten it with { strictMultiplicity: true } to renderSpec).

When data crosses SQL ↔ JS

You author one spec; the optimizer's substrate chooser decides where each relation lives and keeps data where it already is — SQL marks stay in the database (no materialize round-trip), and a few operators (force, treemap in SQL; dag, fitText in JS) run in whichever substrate suits them, inserting a materialize/inject only at a genuine boundary. What pins the choice:

  • Raw SQL (derive/fromQuery/filter/{ sql }) keeps that relation in SQL.
  • JS data (fromRows/fromColumnar) can go either way — the chooser materializes it into the backend when a downstream operator wants SQL.

You rarely think about this; it matters only when you're reasoning about performance or writing an operator. The Debugging tools show the chosen substrate per operator (the Graph tab colors each node by sql / js / framework).
