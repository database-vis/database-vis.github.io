Debugging

DVL's debugging story is relational: the pipeline's own execution is recorded as tables, so you inspect a render the same way you inspect any data — with queries, or with plots.

The DuckDbSink and the dvl_* tables

Pass a DuckDbSink to renderSpec and every compile pass, operator run, SQL statement, cell write, and solver iteration is recorded into dvl_* tables — then explore them with SQL, or with DVL's own self-visualizations (spanTimeline, sqlProfile, convergenceTrace, convergenceResidual, entityCounts, erDiagram).

Self-visualizations

The named self-visualizations are DVL specs over the dvl_* tables:

  • spanTimeline — compile passes and operator runs over time
  • sqlProfile — the recorded SQL statements
  • convergenceTrace, convergenceResidual — solver iterations
  • entityCounts — recorded entities by kind
  • erDiagram — the schema as an ER-style diagram
