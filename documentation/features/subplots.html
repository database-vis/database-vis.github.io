Subplots

Subplots are ad-hoc, heterogeneous panels: each panel hosts its own child plots — its own data, marks, encodings — side by side on one canvas. The whole feature is one idea taken seriously: panels are rows. Each panel is a row of a virtual table, the panel container is an ordinary rect plot over that table, and arranging panels is the same problem as arranging marks — solved by the same layouts. There is no separate subplot engine to learn; everything you know about layouts, scales, and adornments transfers.

Why that's worth understanding:

  • One layout vocabulary. eqX splits marks into columns, so it splits panels into columns. bandX's paddingInner opens gaps between bars, so it opens gutters between panels. Any layout that can place a rect can place your dashboard.
  • Composition is free. Panels share the canvas's scale machinery: bind the same quantity in two panels and the domains union, the frames align, and the redundant interior axis merges away — no manual axis alignment.
  • Heterogeneity is the point. Facets repeat ONE spec over partitions of ONE dataset; subplots compose DIFFERENT specs — a scatter beside a bar chart beside a node-link diagram.

subplot() vs subplots()

Two calls, two orthogonal roles:

Call Role Cardinality
v.subplot() content — appends one row { i: k } to the node's virtual panels table, hosts a child in that row's rect, and returns the child's builder (.dot(...), .bar(...), …) once per panel
v.subplots({ ...channels }) arrangement — binds channels (usually a layout's .pick()) on the shared container plot, over the whole panels table zero or one call

subplot() without subplots() is legal: with no layout bound, the panels' extents are holes and the constraint solver derives the split (two panels → half/half). So subplots() is an override — reach for it when you want a specific arrangement, not to make panels work at all.

The virtual panels table

The table subplots() binds over is real — it just isn't yours. It is a synthesized values relation with one row per subplot() call:

Column Type Value
i number the panel index — 0-based, in subplot() call order
strip_label string present only when v.strip([...labels]) injected labels (one per panel)

That is the whole schema, and it is exactly what a layout argument can name: eqX("i"), eqY("i"), bandX("i"). Channels not produced by the layout can still be bound as literals — a literal broadcasts to every panel row (e.g. one fixed y/h band for all panels).

Choosing a layout

The container is a rect, so any {x, w} / {y, h} producer arranges the panels.

Columns — eqX("i") splits the width equally:

// two panels over the panel-index virtual table; both encode body_mass on y, so the
// solver UNIONS that scale and the interior y-axis merges away — ONE axis speaks for both.
const v = canvas(720, 300);
v.subplot().dot("penguins", { x: linear("bill_length_mm"), y: linear("body_mass_g") });
v.subplot().dot("penguins", { x: linear("flipper_length_mm"), y: linear("body_mass_g") });
v.subplots({ ...eqX("i").pick("x", "w") });
return v;

Rows — same spec, swap the axis pair (eqY gives {y, h}):

// stacked rows instead of columns: the X/Y pair convention applies to panels too.
const v = canvas(720, 460);
v.subplot().dot("penguins", { x: linear("bill_length_mm"), y: linear("body_mass_g") });
v.subplot().dot("penguins", { x: linear("flipper_length_mm"), y: linear("body_mass_g") });
v.subplots({ ...eqY("i").pick("y", "h") });
return v;

Gutters — bandX treats panel indices as a categorical band scale, so paddingInner opens space between panels exactly as it opens space between bars:

// bandX over the panel index: paddingInner is the panel gutter.
const v = canvas(720, 300);
v.subplot().dot("penguins", { x: linear("bill_length_mm"), y: linear("body_mass_g") });
v.subplot().dot("penguins", { x: linear("flipper_length_mm"), y: linear("body_mass_g") });
v.subplots({ ...bandX("i", { paddingInner: 0.15 }).pick("x", "w") });
return v;

Mixed — a layout on one axis, pixel literals on the other (a literal broadcasts to every panel):

// panels in a fixed-height strip: eqX solves x/w per panel; y/h are one pixel band.
const v = canvas(720, 300);
v.subplot().dot("penguins", { x: linear("bill_length_mm"), y: linear("body_mass_g") });
v.subplot().dot("penguins", { x: linear("flipper_length_mm"), y: linear("body_mass_g") });
v.subplots({ ...eqX("i").pick("x", "w"), y: px(30), h: px(220) });
return v;

Heterogeneous panels

Each subplot() is an independent host — different tables, different marks. Titles ride the same virtual table: v.strip([...]) injects strip_label and draws one label per panel (v.strip() with no argument labels by i).

// a scatter beside a bar chart — two datasets, two mark types, one canvas.
db.fromRows("budgets", [
  { category: "Eng", amount: 420 }, { category: "Mkt", amount: 210 }, { category: "Sales", amount: 180 },
]);
const v = canvas(720, 320);
v.subplot().dot("penguins", { x: linear("bill_length_mm"), y: linear("body_mass_g"), fill: ordinal("species") });
v.subplot().bar("budgets", { ...bandX("category").pick("x", "w"), ...barY("amount", { zero: true }).pick("y", "h"), fill: "#4e79a7" });
v.subplots({ ...eqX("i").pick("x", "w") });
v.strip(["Penguins", "Budget"]);
return v;

Subplots vs facets

Both draw side-by-side panels; they answer different questions.

Subplots Facets / nesting
Panels come from explicit subplot() calls (a virtual index table) the rows of a real table + a nest key
Content per panel anything — each panel is authored separately ONE spec, repeated over the key's partitions
Panel count fixed by the spec driven by the data
Use it for dashboard composition, before/after pairs, mixed chart types small multiples of the same chart

If your panels are "the same chart per species", nest. If they are "these two charts, together", subplot.
