Layouts

Operators that compute arrangements — usually multi-output (.pick("x", "w")). The three classes: produce (data → normalized positions: bandX, barY, stackY, eqX, force, treemap, proportional*), refine (positions → positions, post-placement: fitText, grid-stack), measure (positions → sizes, feeding the demand loop). Reference: layouts/.

Each 1-D producer is an X/Y pair — bandX/bandY, barX/barY, stackX/stackY, pitchX/pitchY, eqX/eqY — so a vertical bar (bandX + barY) mirrors to a horizontal one (bandY + barX) by swapping the axis; eqX splits columns, eqY rows.

// multi-output layouts spread into channels with .pick(): bandX gives {x, w}, stackY {y, h}.
const stacks = db.table("penguins").groupby(["species", "island"], { n: count() });
const v = canvas(640, 340);
v.rect(stacks, {
  ...bandX("species", { paddingInner: 0.2 }).pick("x", "w"),
  ...stackY({ weight: "n", group: "species", order: "island" }).pick("y", "h"),
  fill: ordinal("island"),
}).axis("x").legend("fill");
return v;

Binding with .pick()

Most layouts are multi-output: .pick() selects outputs and spreads them into the channel object. ...bandX("cat").pick("x", "w") binds the layout's x and w outputs to the mark's x and w channels; the object form renames — .pick({ x1: "x", y1: "y" }) binds the layout's x output to the mark's x1. Producer outputs are normalized [0, 1]; the frame's affine turns them into pixels (see The frame).

The three classes

Classes are told apart by data-flow direction — what a layout reads and what it writes:

  • Produce — data → normalized positions. The common case: a train/apply operator reduces the marks to a fit, then joins the fit back per row to emit normalized [0, 1] channel columns (bandX, barY, stackY, eqX, proportional*). Iterative producers (force, treemap) emit the same normalized outputs from a custom plan — in SQL, a recursive query.
  • Refine — placed pixels → adjusted pixels. Runs after placement, reading the marks' already-solved geometry: fitText reads each label's box and the string it draws to solve a font size; grid-stack packs items into the content box. Refine outputs are pixels — they never pass back through the affine.
  • Measure — placed geometry → size demands. The reverse direction: a measure reports how much room a plot or label needs, feeding the constraint solver's demand loop (an axis gutter widens to fit its measured labels; a panel grows to its content). Not exposed in the authoring API — the engine wires the built-in providers itself.

Together they close the frame's fixpoint: produce emits normalized coordinates, the affine places them, refine nudges the placed pixels, and measure reads the result back into size demands that re-solve the arrangement until it settles. The reference page also describes a fourth, orthogonal resolver class — a reduction riding a via() join, not a placement stage (see layouts/).

X/Y pairs

Every 1-D producer comes in an X and a Y form with mirrored outputs, so swapping the pair transposes the chart:

Pair Outputs Arrangement
bandX / bandY x, w / y, h categorical → evenly spaced bands
barX / barY x, w / y, h value → a bar from the baseline out/up
stackX / stackY x, w / y, h { weight, group, order } → stacked segments per group
pitchX / pitchY x, w / y, h fixed-pitch bands, one per row (legend items, list rows)
eqX / eqY x, w / y, h distinct keys → equal split (eqX columns, eqY rows)

A vertical bar chart is bandX + barY; the horizontal mirror is bandY + barX. The 2-D producers (force, treemap) and the refine-class layouts are cataloged in the reference.
