Via: cross-plot and cross-table references

via() is DVL's one verb for borrowing values through a relational join: a layer names a referent, a join path, and the outputs it wants; the compiler emits the joins. Everything else — what arrives, when it resolves, and which rows the layer even has — is decided by two classifications, not by extra API: what the referent is (a plot or a table) and where the consuming layer lives (top-level or nested inside a panel template).

// the shape: referent.via(path…, opts?).pick({ channel: output, … })
np.via(eq('follows.src', 'members.id')).pick({ x1: 'x', y1: 'y' })

Paths are the same join-path vocabulary as everywhere else — eq/lt/…, on('col'), fk('table'), multi-hop path(...) — resolved once against the FK catalog. A bare string is the same-column shorthand: via("id") ≡ via(eq("id", "id")), mixing freely with other steps (via("id", fk("islands"))) — and it works at every path site that takes this vocabulary (via, nest's on:, a layout's aux on:), since the desugar happens where paths resolve, not in any one API entry.

The referent axis: plot vs table

Plot referent — np.via(…) Table referent — db.table('t').via(…)
Borrows the referent's solved channels the table's raw columns
Arrives in pixel space (post-affine — never re-placed) data space (free to feed a scale, train a domain, be placed)
Joins the referent plot's placed marks, after placement a table leaf, in the pre-affine chain
Typical use edge endpoints, connectors, annotations on placed marks denormalization-free attribute lookup across an FK

Same verb, same pick, same resolution grammar — the borrowed column's Space carries the difference, so the compiler knows a pixel borrow must never pass through the affine again while a data borrow still may. This is the type system doing the deciding, not a mode flag.

// both timings in one figure: every bird LINKS to its species centroid. x1/y1 borrow the
// dot's own solved position (a single-match via on the tuple key); x2/y2 borrow the
// CENTROID — a multi-match via reduced by resolve: 'avg'. All four arrive in pixels.
const p = db.table("penguins").withKey("id");
const v = canvas(640, 360);
const dots = v.dot(p, { x: linear("bill_length_mm"), y: linear("body_mass_g"), fill: ordinal("species"), title: field("id") });
v.link(p, {
  ...dots.via(eq("id", "id")).pick({ x1: "x", y1: "y" }),
  ...dots.via(eq("species", "species"), { resolve: "avg" }).pick({ x2: "x", y2: "y" }),
  stroke: "#bbb", strokeOpacity: 0.5,
});
return v;

The consumer axis: top-level vs nested

Where the consuming layer is authored decides which rows it has and which panel each row belongs to — the receiver is the declaration:

Layer Rows Panel identity Authoring
top-level the whole relation none v.link('follows', { …vias… })
top-level + frame: the whole relation ASSIGNED — one reference names the owner panel v.link('follows', { …vias… }, { frame: srcVia })
nested the panel's sub-database σ implied by membership panel.link('follows', { …vias… })
nested + scope: 'union' σ, any-path membership one row per touching panel panel.link('follows', { …vias… }, { scope: 'union' })

The nested rows come from taking the partition seriously: a template cell scopes the whole database, so inside a panel a table name denotes its restriction, derived through the declared FK paths (conjoined by default — a row belongs where all its references land; a cross-panel edge simply isn't in either panel's data). No on, no reserved channel: nesting the layer is the whole declaration. Rows the partition drops are reported as data (the layer's sigmaDropped diagnostics relation), never silently.

frame: is the explicit opt-out — global rows, each assigned a panel by the named reference (the via handle, or the predicate itself). It is an opts key, not a channel, so exactly one assignment per layer is structurally guaranteed. The old pick({ cell: 'dvl_cell' }) spelling still works with a deprecation warning.

Timing: one declared join, read at two times

A single FK can serve two roles, resolved at different points in the pipeline:

  • the frame reads it early — before layout — because the referent's panel (dvl_cell) is known as soon as the referent is nested;
  • the positions read it late — post-affine — because pixels exist only after placement.

So frame: src plus src.pick({ x1: 'x', y1: 'y' }) is one relationship, read twice. Cross-node references (a top-level layer borrowing from a plot nested in a later-compiled panel) work because the compiler patches late-bound plot references after the full walk — see Architecture → late-bound plot references.

Multiplicity and resolutions

A via join that can match several referent rows per mark must declare how the match set reduces to one value — a bare AggFn, an ordered positional form, a named expression over the two-sided (m, f) row, or an opaque JS function (inline, or a named reducer registered in registries.reducers). Provably multi-match joins without a resolution are a build error, never a silent fan-out; the lenient default for unprovable joins tightens to an error under { strictMultiplicity: true }. The full grammar with examples is in Cross-mark & cross-table references; the design rationale (input schema, naming of computed outputs, the emitted SQL) is in design/CELL_KEY_VIA.html §3b–§3c.

Choosing, quickly

  • Draw edges between placed marks in one frame → top-level layer + position vias.
  • Same, but each edge should belong to a panel (per-panel stats, clipping) → add frame:.
  • Small multiples where each panel shows only its own subgraph → nest the layer; done.
  • Each panel shows its local subgraph including edges that touch it → nested + scope: 'union'.
  • Look up an attribute across an FK without denormalizing → db.table(...).via(...), data space.
  • Aggregate over a one-to-many relationship → any of the above + { resolve: … }.

Turning borrowed endpoints into boundary attachments: route()

A position via hands back the referent's center in pixels. For a node-link edge you usually want the endpoint on the node's boundary instead — the arrowhead should meet the circle, not bury itself in the middle. route() is the idiomatic layout for that: it owns the two endpoint borrows and outputs the final clipped endpoints.

v.link("edges", {
  ...route({ src: nodes.via(eq("edges.src", "nodes.id")),
             dst: nodes.via(eq("edges.dst", "nodes.id")) }).pick("x1", "y1", "x2", "y2"),
  stroke: "#999",
});

Pass the endpoint via handles directly (not .pick()ed), pick all four channels, and each endpoint lands on the referent's boundary nearest the far end — a dot clips on its circle, a rect/bar/image on its rectangle edge. This replaces manually resolving a center (the old { resolve: { cx: { fn: 'avg', arg: exprf.x + w/2 } } } boilerplate): route() derives the center from the box edges and clips to the true boundary in one step. arrow composes the same way — its head then lands on the target boundary. route() is post-affine and segment-only, so it does not apply to curved bezier links yet.
