Channels

A channel is a named input to a mark — x, y, x1, fill, r, text, dx… Every mark declares which channels it accepts and each channel's kind (positional, pixel, color, shape, text, number — see Marks). What you bind to a channel is a value in one small channel-value language. This page maps that language: the ways to specify a channel, what each one produces, and where and when it resolves.

Whatever you write, the compiler lowers it to a single union — the Expr type in ir/expr.ts:

literal · local-column · layout-output · table-ref · bound-ref · agg · binary · var

The authoring surface (ChannelValue) is a little broader than Expr — it also accepts scale specs, layout .pick() handles, and via .pick() handles, which the builder resolves into operators plus an Expr reference. But everything ends as that union, and it lowers once to both substrates (SQL in the database, JS in the browser) — you never pick a backend.

type ChannelValue =
  | ScaleSpec | ColumnBinding | DatumBinding   // scales
  | PendingOutput                              // a layout / via .pick() output
  | Expr                                       // field(), binary(), literal(), px.*(), …
  | number | string | boolean                  // bare literals
  | null;                                      // an explicit hole (unbound)

Two questions every channel value answers

Two cross-cutting properties decide how a value behaves; internalizing them explains the whole reference below.

Space — data or pixel?

Space Meaning Feeds a scale? Placed by the frame?
data a value in the data domain — a raw column, a constant, a table borrow. Pre-affine. yes yes
pixel a value already in canvas coordinates — a positional literal, precomputed geometry, a borrowed solved position. Post-affine. no no — drawn as-is, never re-placed

Resolution — when is it known? A value's resolution is the max over its leaves, ordered literal < data < solver:

  • literal — a constant, known at author time.
  • data — computed by a query over your tables (a column, a table borrow, an aggregate).
  • solver — produced by the constraint solver (layout outputs, borrowed pixel positions) — known only after placement.

(Arity is the quieter third axis: a constant is scalar, a column is per-row; arithmetic broadcasts scalar over per-row.)

Frame variables (var — the solver's fixpoint cells for extents, params, ranges) are an internal form you rarely author directly; they surface in axis internals and params.

Data: columns and constants

field("c") reads a column untransformed (data space, per-row). Bare literals are constants — and a bare string is the literal string, never a column: text: "island" labels every mark with the word island; text: field("island") reads the column. A bare number means what its channel declares: on positional channels (x, y, x1…) it is a data value through the axis's scale (px(v) for pixels); on pixel-size channels (r, w, dx…) it is pixels; fill: "#4e79a7" is the literal colour.

// literal vs scale vs column: x/y are SCALE bindings; r: 4 and fill: '#4e79a7' are
// LITERALS (a bare string is never a column read); the label reads a column with field().
const v = canvas(640, 320);
const x = linear("bill_length_mm"), y = linear("body_mass_g");
v.dot("penguins", { x, y, r: 4, fill: "#4e79a7", fillOpacity: 0.6 });
v.text("penguins", { x, y, text: field("island"), dy: -8, fontSize: 8, fill: "#888" });
return v;

Scales: a value through a function

A scale trains a domain from data and maps each row into an output space. Three forms: linear("c") fits to a column immediately; const x = linear(); x("c") is deferred and bound per channel — binding the same scale object twice declares "the same quantity", so the domains union and the layers share a frame; x(4000) is a datum, a literal placed in the scale's domain. See Scales for the families and domain treatments.

// ONE scale object bound across two relations declares "the same quantity": the domains
// UNION and both layers share a frame — the species means land exactly on the cloud.
const bySpecies = db.table("penguins").groupby(["species"], { bill: avg("bill_length_mm"), mass: avg("body_mass_g") });
const v = canvas(640, 360);
const x = linear(), y = linear();
v.dot("penguins", { x: x("bill_length_mm"), y: y("body_mass_g"), fill: "#cfd3dc" });
v.dot(bySpecies, { x: x("bill"), y: y("mass"), r: 6, fill: ordinal("species") })
 .axis("x").axis("y").legend("fill");
return v;

Layout outputs: arrangements via .pick()

A layout is an operator that computes an arrangement — usually multi-output. Spread its outputs into channels with .pick(); the values are solver-space pixels. See Layouts.

// multi-output layouts spread into channels with .pick(): bandX gives {x, w}, stackY {y, h}.
const stacks = db.table("penguins").groupby(["species", "island"], { n: count() });
const v = canvas(640, 340);
v.rect(stacks, {
  ...bandX("species", { paddingInner: 0.2 }).pick("x", "w"),
  ...stackY({ weight: "n", group: "species", order: "island" }).pick("y", "h"),
  fill: ordinal("island"),
}).axis("x").legend("fill");
return v;

Via: borrowed values

via() borrows values across a relational join path. The referent decides the space: a plot referent borrows solved channels → pixels (never re-placed); a table referent borrows raw columns → data (free to feed a scale). A join that can match several rows per mark declares a resolution ({ resolve: "avg" }, a positional form, an expression, or a JS reducer). Full mechanism: Via.

// PLOT referent + a resolution, live: each species label sits at the CENTROID of its dots —
// the via matches MANY dots per species; resolve: 'avg' reduces every picked channel
// independently. The borrowed values are solved pixels, never re-placed.
const species = db.table("penguins").groupby(["species"], {});
const v = canvas(640, 360);
const dots = v.dot("penguins", { x: linear("bill_length_mm"), y: linear("body_mass_g"), fill: ordinal("species"), fillOpacity: 0.5 });
v.text(species, {
  ...dots.via("species", { resolve: "avg" }).pick({ x: "x", y: "y" }),
  text: field("species"), fontSize: 12, fill: "#333", anchor: "middle",
});
return v;

Arithmetic: combining expressions

binary(op, left, right) combines any two channel expressions — columns, literals, borrows — with + - * / %. It lowers once to both substrates through the shared ScalarExpr layer, and the result's space is dimensionally checked (a pixel offset on a placed value; a data offset on a borrowed column). A via/aggregate nested inside arithmetic is lifted into its own join automatically, so y: borrowed + 5 works. The cexpr`…` tagged template is sugar for the same tree — bare names are columns, ${…} splices a channel expression: cexpr`x + dx` ≡ binary("+", field("x"), field("dx")).

// ARITHMETIC channels: each arrow's head is a data expression — cexpr`…` builds the tree (bare
// names are columns), compiling once to both substrates (SQL + JS) via the ScalarExpr lowering.
// Feeding it through the SAME scale as the tail (x()/y()) puts both endpoints in one domain — one coupled axis.
const flow = db.derive("flow", `
  SELECT bill_length_mm AS x, body_mass_g AS y,
         (bill_depth_mm - 17) * 0.3 AS dx, (flipper_length_mm - 200) * 40 AS dy
  FROM penguins`);
const v = canvas(640, 360);
const x = linear(), y = linear();
v.arrow(flow, {
  x1: x("x"), y1: y("y"),
  x2: x(cexpr`x + dx`),
  y2: y(cexpr`y + dy`),
  headType: "barb", headSize: 5, stroke: "#4e79a7", strokeOpacity: 0.6,
}).axis("x").axis("y");
return v;

Pixels: explicit geometry

Pixel intent is always spelled: px(12) on any channel is 12 screen pixels, and px("col") reads a column as a pixel coordinate or length — bypassing the frame placement. This is how a table can size its own boxes: the engine reads pixel channels as canvas coordinates and never runs them through a scale. (The axis-explicit readers px.x / px.y / px.w / px.h remain for expression positions, where types are checked before any channel is known.)

// precomputed geometry: this table already carries pixel box edges; px.* reads a column as a
// CANVAS coordinate/length, skipping the frame affine — so each rect places itself.
db.fromRows("boxes", [
  { label: "A", x: 40, y: 60, w: 150, h: 170 },
  { label: "B", x: 210, y: 60, w: 150, h: 170 },
  { label: "C", x: 380, y: 60, w: 180, h: 170 },
]);
const v = canvas(600, 290);
v.rect("boxes", { x: px.x("x"), y: px.y("y"), w: px.w("w"), h: px.h("h"), fill: ordinal("label") }).legend("fill");
return v;

Channel kinds: what a mark accepts

Marks type-check what you bind. Each channel has a kind — positional-x/y, pixel, color, shape, text, number — and the binding must agree:

  • a scale or column goes on positional/color channels; a shape/text channel takes a literal or a column, not a scale output;
  • on a color channel, a continuous scale is a sequential ramp and ordinal is a categorical palette (color is an output space, not a scale family);
  • the bare-string rule holds on every channel, text included — text: "n" is the literal, text: field("n") is the column;
  • an unbound channel is a hole (null / omitted) — the mark uses its draw-time default.

Per-mark channel lists are in the Marks reference; the concepts they build on — scales, layouts, the frame, via — each have their own page under Features.
