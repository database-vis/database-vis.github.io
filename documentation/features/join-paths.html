Join paths

"Relate this relation to that one, on these columns, possibly through intermediates" is one concept — a Path — and every join site in DVL consumes it: nest keys, cross-mark and cross-table via, layout aux inputs (force's edges). Whatever the site, you describe the relationship once, in one vocabulary, and the compiler emits the joins.

One vocabulary

One constructor vocabulary:

eq('penguins.island', 'islands.island')   // an explicit predicate (also lt/le/gt/ge/neq)
on('island')                              // same-column eq shorthand
fk('islands')                             // a named FK hop — expands against the catalog
path('friends', 'penguins', 'islands')    // a multi-hop chain of FK hops
// …and OMITTING the path infers the unique FK between source and target

The constructors range from fully explicit to fully inferred: eq (and its comparison siblings lt/le/gt/ge/neq) spells out a predicate column by column; on('island') is the same-column eq shorthand; fk('islands') names a foreign key and expands it against the catalog; path(...) chains FK hops through intermediate tables; and omitting the path entirely asks the resolver to infer the unique FK between source and target.

One Path, many sites — policy, not type

Paths resolve once (against the database's FK catalog — see Data); what varies per site is policy, not type: a nest key must resolve to all-eq conditions (it feeds a partition key), while a via admits any predicate — a range join just matches a set, which is what resolutions are for. The resolver also judges multiplicity: a hop that lands on a known key (an FK's target) is provably single-match; a non-eq hop is provably multi; multi without a declared resolution is a build-time error, never a silent fan-out.

Multiplicity

The resolver's multiplicity judgment is what keeps joins honest. It classifies each resolved path:

  • a hop that lands on a known key (an FK's target) is provably single-match;
  • a non-eq hop (a range predicate like gt/lt) is provably multi-match;
  • a provably multi-match join without a declared resolution is a build-time error — never a silent fan-out.

This is why declaring FKs pays off: they are the facts the judgment reads (see Data for the catalog).

Example: paths at a via site

// a node-link diagram: force places the members; route() OWNS both endpoint borrows and
// clips each edge to the node BOUNDARY nearest the far end. The `eq('follows.src','members.id')`
// predicates ARE the join paths — `on('col')` (same-column eq), `fk('table')` (named hop), and
// `path(...)` (multi-hop) are the shorthands the section lists, resolved against the same FK catalog.
const v = canvas(640, 420);
const nodes = v.dot("members", { ...force("follows").pick("x", "y"), r: 8, fill: ordinal("community") });
v.link("follows", {
  ...route({ src: nodes.via(eq("follows.src", "members.id")),
             dst: nodes.via(eq("follows.dst", "members.id")) }).pick("x1", "y1", "x2", "y2"),
  stroke: "#bbb", strokeOpacity: 0.6,
});
return v;

Because route() owns the endpoint joins, each edge lands on the node's boundary (a dot clips on its circle) — not its center — so arrowheads and links meet the mark cleanly.
